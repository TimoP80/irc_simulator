üìñOverview

This documentation describes the JSON format used by Station V - Virtual IRC Simulator for storing and exchanging data. The format has evolved significantly to support advanced features like time-of-day synchronization, multi-language support, and enhanced user customization.

Version 1.13.12 Features: This documentation covers the latest format including time-of-day synchronization, per-language fluency support, dominant language channels, and enhanced writing styles.

Key Data Types

  • User Objects: Virtual AI users with personalities, language skills, and writing styles
  • Channel Objects: IRC channels with topics, users, messages, and operators
  • Message Objects: Individual chat messages with metadata
  • Configuration Objects: Application settings and preferences

üë§User Object Format

The User object represents a virtual AI user in the simulation. It contains all the information needed to generate realistic, in-character responses.

Basic Structure

{ "nickname": "string", "status": "online" | "away", "personality": "string", "languageSkills": { /* Language Skills Object */ }, "writingStyle": { /* Writing Style Object */ } }

Property Details

Property Type Required Description
nickname string ‚úÖ Yes Unique identifier for the user (3-15 characters, alphanumeric + underscore/hyphen)
status string ‚úÖ Yes Current online status: "online" or "away"
personality string ‚úÖ Yes Detailed personality description (used by AI for character consistency)
languageSkills object ‚úÖ Yes Language abilities and fluency levels (see Language Skills Format)
writingStyle object ‚úÖ Yes Communication preferences (see Writing Styles Format)

Example User Object

{ "nickname": "nova", "status": "online", "personality": "A very curious and tech-savvy individual, loves discussing new gadgets and programming. Always eager to learn and share knowledge.", "languageSkills": { "languages": [ { "language": "English", "fluency": "native", "accent": "American" }, { "language": "Python", "fluency": "advanced", "accent": "Code comments" } ] }, "writingStyle": { "formality": "informal", "verbosity": "neutral", "humor": "witty", "emojiUsage": "medium", "punctuation": "standard" } }

#Ô∏è‚É£Channel Object Format

Channel objects represent IRC channels where users can communicate. They include topic, user lists, message history, and operator permissions.

Basic Structure

{ "name": "string", "topic": "string", "users": [ /* Array of User Objects */ ], "messages": [ /* Array of Message Objects */ ], "operators": [ /* Array of nicknames */ ], "dominantLanguage": "string" // Optional }

Property Details

Property Type Required Description
name string ‚úÖ Yes Channel name (must start with #, e.g., "#general")
topic string ‚úÖ Yes Channel topic/description
users array ‚úÖ Yes Array of User objects currently in the channel
messages array ‚úÖ Yes Array of Message objects (chat history)
operators array ‚úÖ Yes Array of user nicknames with operator privileges
dominantLanguage string ‚ùå No New in v1.13.10: Explicit dominant language for the channel (e.g., "Finnish", "English")

Example Channel Object

{ "name": "#tech-talk", "topic": "Discussing the latest in technology and software development", "users": [ { /* User object for nova */ }, { /* User object for rex */ } ], "messages": [ { "id": 1, "nickname": "system", "content": "You have joined #tech-talk", "timestamp": "2025-01-21T14:30:00.000Z", "type": "system" } ], "operators": ["nova", "rex"], "dominantLanguage": "English" }

üí¨Message Object Format

Message objects represent individual chat messages with full metadata for proper display and processing.

Basic Structure

{ "id": "number", "nickname": "string", "content": "string", "timestamp": "string (ISO 8601)", "type": "string", "command": "string", // Optional "target": "string" // Optional }

Property Details

Property Type Required Description
id number ‚úÖ Yes Unique message identifier (typically timestamp-based)
nickname string ‚úÖ Yes Nickname of the user who sent the message
content string ‚úÖ Yes The actual message content
timestamp string ‚úÖ Yes ISO 8601 timestamp of when the message was sent
type string ‚úÖ Yes Message type: system, user, ai, pm, action, notice, topic, kick, ban, join, part, quit
command string ‚ùå No IRC command type (for special message types)
target string ‚ùå No Target user or channel (for commands)

Example Message Objects

// Regular AI message { "id": 1705845000000, "nickname": "nova", "content": "Hey everyone! Just discovered this amazing new JavaScript framework.", "timestamp": "2025-01-21T14:30:00.000Z", "type": "ai" } // System message { "id": 1705845000001, "nickname": "system", "content": "You have joined #tech-talk", "timestamp": "2025-01-21T14:30:01.000Z", "type": "system" } // Action message (/me command) { "id": 1705845000002, "nickname": "rex", "content": "is debugging the server configuration", "timestamp": "2025-01-21T14:30:02.000Z", "type": "action", "command": "me" }

‚öôÔ∏èConfiguration Format

The configuration object stores application settings and user preferences. It supports both text-based and object-based storage formats.

Basic Structure

{ "currentUserNickname": "string", "virtualUsers": "string", // Text format "channels": "string", // Text format "simulationSpeed": "string", "aiModel": "string", "typingDelay": { /* Typing Delay Object */ }, "userObjects": [ /* Array of User Objects */ ] // Optional }

Property Details

Property Type Required Description
currentUserNickname string ‚úÖ Yes Your display name in the simulation
virtualUsers string ‚úÖ Yes Text representation of virtual users (for easy editing)
channels string ‚úÖ Yes Text representation of channels (for easy editing)
simulationSpeed string ‚úÖ Yes Simulation speed: "fast", "normal", "slow", or "off"
aiModel string ‚úÖ Yes AI model identifier (e.g., "gemini-2.5-flash")
typingDelay object ‚úÖ Yes Typing delay configuration (see below)
userObjects array ‚ùå No New in v1.13.11: Full User objects for proper persistence

Typing Delay Object

{ "enabled": "boolean", "baseDelay": "number", // milliseconds (default: 500) "maxDelay": "number" // milliseconds (default: 15000, range: 1000-30000) }

Example Configuration

{ "currentUserNickname": "you", "virtualUsers": "nova: A curious tech-savvy individual...\nseraph: Calm and wise...", "channels": "#general: General discussion | English\n#tech-talk: Technology discussions | English", "simulationSpeed": "normal", "aiModel": "gemini-2.5-flash", "typingDelay": { "enabled": true, "baseDelay": 500, "maxDelay": 15000 }, "userObjects": [ { /* Full User object for nova */ }, { /* Full User object for seraph */ } ] }

üåçLanguage Skills Format

Language skills define what languages a user speaks and their fluency levels. Station V supports two formats for backward compatibility.

Format 1: Per-Language Format Recommended

{ "languages": [ { "language": "string", "fluency": "beginner" | "intermediate" | "advanced" | "native", "accent": "string" // Optional } ] }

Fluency Level Options

beginner // Basic understanding, simple phrases intermediate // Good understanding, can hold conversations advanced // Fluent, can discuss complex topics native // Native-level fluency, perfect command

Format 2: Legacy Format Deprecated

{ "fluency": "beginner" | "intermediate" | "advanced" | "native", "languages": ["string"], "accent": "string" // Optional }
Migration Notice: The per-language format is recommended for new projects. The legacy format is still supported for backward compatibility but may be deprecated in future versions.

Per-Language Format Example

{ "languages": [ { "language": "English", "fluency": "native", "accent": "American" }, { "language": "Finnish", "fluency": "intermediate", "accent": "Helsinki" }, { "language": "Python", "fluency": "advanced", "accent": "Clean code style" } ] }

Multi-Language User Example

{ "languages": [ { "language": "English", "fluency": "native", "accent": "British" }, { "language": "Spanish", "fluency": "advanced", "accent": "Mexican" }, { "language": "French", "fluency": "intermediate", "accent": "Parisian" }, { "language": "Japanese", "fluency": "beginner", "accent": "Tokyo" }, { "language": "JavaScript", "fluency": "advanced", "accent": "ES6+ style" } ] }

Legacy Format Example

{ "fluency": "native", "languages": ["English", "Finnish"], "accent": "Multilingual" }

‚úçÔ∏èWriting Styles Format

Writing styles control how users communicate, affecting message length, tone, and formatting.

Basic Structure

{ "formality": "very_informal" | "informal" | "neutral" | "formal" | "very_formal", "verbosity": "very_terse" | "terse" | "neutral" | "verbose" | "very_verbose", "humor": "none" | "dry" | "sarcastic" | "witty" | "slapstick", "emojiUsage": "none" | "low" | "medium" | "high" | "excessive", "punctuation": "minimal" | "standard" | "creative" | "excessive" }

Property Details

Property Values Description
formality very_informal, informal, neutral, formal, very_formal Level of formality in communication
verbosity very_terse, terse, neutral, verbose, very_verbose Length and detail of messages
humor none, dry, sarcastic, witty, slapstick Humor style and frequency
emojiUsage none, low, medium, high, excessive Frequency of emoji usage
punctuation minimal, standard, creative, excessive Punctuation style and frequency

Available Language Options

The language selector in the user editor supports the following languages:

English, Finnish, Spanish, French, German, Italian, Portuguese, Russian, Chinese, Japanese, Korean, Arabic, Dutch, Swedish, Norwegian, Danish, Polish, Czech, Hungarian, Greek, Turkish, Hebrew, Hindi, Thai, Vietnamese, Indonesian, Malay, Tagalog, Other (specify in accent field)
Language Support: The "Other" option allows you to specify any custom language in the accent field. This is useful for programming languages, constructed languages, or regional dialects not in the predefined list.

Complete Option Reference

Formality Options

very_informal // Very casual, uses slang, contractions informal // Casual, friendly, conversational neutral // Balanced, neither formal nor informal formal // Professional, proper grammar very_formal // Very formal, academic, official tone

Verbosity Options

very_terse // Very brief, minimal words terse // Brief, concise messages neutral // Moderate length, balanced verbose // Detailed, longer messages very_verbose // Very detailed, comprehensive messages

Humor Options

none // No humor, serious tone dry // Subtle, understated humor sarcastic // Sarcastic, ironic humor witty // Clever, quick-witted humor slapstick // Physical, exaggerated humor

Emoji Usage Options

none // No emojis used low // Occasional emojis medium // Regular emoji usage high // Frequent emoji usage excessive // Very frequent, emoji-heavy

Punctuation Options

minimal // Minimal punctuation standard // Standard punctuation creative // Creative use of punctuation excessive // Heavy use of punctuation

Complete Writing Style Examples

// Very formal, verbose academic { "formality": "very_formal", "verbosity": "very_verbose", "humor": "none", "emojiUsage": "none", "punctuation": "standard" } // Casual, funny social media user { "formality": "very_informal", "verbosity": "neutral", "humor": "slapstick", "emojiUsage": "excessive", "punctuation": "creative" } // Professional, concise business user { "formality": "formal", "verbosity": "terse", "humor": "dry", "emojiUsage": "none", "punctuation": "standard" } // Creative, expressive artist { "formality": "informal", "verbosity": "verbose", "humor": "witty", "emojiUsage": "high", "punctuation": "creative" }

‚úÖValidation Rules

These rules ensure your JSON data is valid and will work correctly with Station V.

User Validation

  • nickname: 3-15 characters, alphanumeric + underscore/hyphen only
  • status: Must be exactly "online" or "away"
  • personality: Non-empty string, recommended 20+ characters
  • languageSkills: Must be valid per-language or legacy format
  • writingStyle: All properties must use valid enum values

Channel Validation

  • name: Must start with #, 2-50 characters total
  • topic: Non-empty string
  • users: Array of valid User objects
  • messages: Array of valid Message objects
  • operators: Array of valid nicknames
  • dominantLanguage: Optional, must be valid language name

Message Validation

  • id: Positive number, should be unique
  • nickname: Non-empty string
  • content: Non-empty string
  • timestamp: Valid ISO 8601 date string
  • type: Must be valid message type
Auto-Validation: Station V automatically validates imported JSON data and provides helpful error messages for invalid formats.
